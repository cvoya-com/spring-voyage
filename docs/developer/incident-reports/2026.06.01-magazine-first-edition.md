## Issue 1 — 401 "Missing or invalid bearer token" on message sends  ⟶ CRITICAL

**Symptom (what you saw).** Agents repeatedly failed to send messages and worked around it. The Managing Editor literally wrote itself a retry note in memory:

> *"Audience Editor dispatch **failed due to 401 error** — retry on next turn … **401 'Missing or invalid bearer token' on all sv.messaging calls** during turn processing inbound message 9cd8c540…"*
> — `…/memory/pending_dispatch_piece1.md`

The raw error (from the agent's MCP sidecar logs), recurring throughout the run:

```
Worker MCP endpoint returned HTTP 401:
{"jsonrpc":"2.0","id":null,"error":{"code":-32001,"message":"Missing or invalid bearer token.","data":null}}
```

Counts in the current container generation (earlier generations were lost with their volumes — see Issue 2): Managing Editor 15, Production Editor 3, others 0. **The 401s cluster on the highest-concurrency agents.** First occurrences ~06:48, i.e. ~5 min into the run — *not* at the 60-min token-expiry mark, so this is a race, not simple expiry.

**Root cause.** The MCP session token used for every `sv.*` tool call (including `sv.messaging.*`) is **per-turn and single-use-scoped**:

- The worker mints one `McpSession` per `(token, agentId, threadId, messageId)` and **revokes it when the turn ends** (`src/Cvoya.Spring.Dapr/Mcp/McpServer.cs` — `IssueSession` / `RevokeSession`; a request whose bearer token isn't in `_sessions` → 401).
- The long-running A2A sidecar writes that per-turn token to a **single file** `…/.spring/bridge/mcp-token`; the CLI-spawned MCP-server process reads it once at startup. The design explicitly **assumes "one CLI spawn = one turn = one token"** (`src/Cvoya.Spring.AgentSidecar/src/mcp-token-store.ts`, citing ADR-0057/ADR-0054). Its own comment notes that a missing/stale file "uses the empty string and **the worker returns 401, which the CLI surfaces as a tool error the model can retry**."

This team runs **persistent agents with `SPRING_CONCURRENT_THREADS=true`** — multiple turns execute at once against that **one shared token file**. Concurrent turns clobber each other's token, and a token revoked at the end of turn A is still held by turn B's in-flight tool call → "Missing or invalid bearer token." The per-turn / single-message token model is fundamentally mismatched with persistent, concurrent-thread agents.

**Impact.** Failed sends → manual retries and resends → duplicate and out-of-order messages → most of the message volume and confusion. (E.g. the Copy Editor resent Piece 1 three times at 07:16 / 07:21 / 07:27 "after the Managing Editor indicated the initial message hadn't come through cleanly.")

## Issue 2 — Agent workspace volumes are wiped on teardown ⟶ no durable memory  ⟶ CRITICAL

**This is the "memory system" gap you suspected.**

Each agent's durable memory (`.claude/projects/**/memory/*.md`) and session transcripts live on a per-agent Podman volume `spring-ws-<agentId>`. The worker **deletes and recreates that volume on every container teardown**:

```
Reclaiming workspace volume spring-ws-75b711e4…           # AgentVolumeManager.ReclaimAsync
Requesting dispatcher to remove volume … → DELETE …/volumes/spring-ws-75b711e4…
Workspace volume … reclaimed            ← deleted (memory + sessions gone)
Requesting dispatcher to ensure volume …
Workspace volume … ensured              ← recreated EMPTY
```

Direct observation: of the 7 agents, only the **3 currently-running** agents still have their `spring-ws-<id>` volume. The Fact-Checker, Copy Editor, and Audience Editor volumes are **gone**. The Copy Editor's memory — which I read ~10 minutes earlier — no longer exists. The reclaim is the **ephemeral "turn done" path** (`EphemeralAgentRegistry.ReleaseAsync` → `volumeManager.ReclaimAsync(agentId)`, ADR-0029) being applied to agentId-keyed volumes that the *persistent* agent depends on.

The agents felt this directly. From the Managing Editor's own post-mortem notes:

> *"the Managing Editor's context window failed to retain incoming messages across multiple turns … Asked the Copy Editor for the Piece 1 copy-edited text **five times**; Copy Editor had to resend repeatedly … Running budget was never updated during the day, so it still showed REPORTING for Piece 1 even after fact-check and copy-edit were complete — creating a false alarm with the Director at assembly."*
> — `…/memory/feedback_context_visibility.md`

Because the workspace is wiped each cycle, `claude --resume` has nothing to resume; the agent restarts cold each turn with only the (small) per-thread actor-channel window as context. There is no durable shared team state — no edition status board — so each agent holds a different, drifting view of where each piece stands. (At 07:36 the Magazine Director was still demanding "direct confirmation … neither has a direct signal from you on the record" for Piece 1, while the Managing Editor's notes already had it "SENT TO PUBLISHER.")

**Secondary defect:** the reclaim sometimes fails — `"Failed to reclaim workspace volume spring-ws-752e7931…; Dispatcher returned 500 … volume may need manual cleanup"` — leaking volumes. 16 orphaned `spring-ws-*` volumes are currently lying around.

## Issue 3 — Sender-identity collapse at the unit address  ⟶ HIGH

**The message you quoted** (Managing Director → Magazine Director):

> *"the Staff Writer has correctly noted they can't distinguish between your messages and mine when both arrive through the unit address. Please send them a brief direct confirmation — from your own address — that 'Seven Made This. Six Are AI.' … is the approved version and Piece 3 is done."*

Two distinct senders — the human overseer ("Managing Director" = Savas) and the unit agent (Magazine Director) — both reach members as `from: unit:75b711e4…`, so recipients can't tell who said what. This is not cosmetic; it corrupted the edit:

> *"A message **appearing to come from the Director (unit:75b711e4…)** reversed the 'Seven' ruling on Piece 3; **the Director later disavowed sending it.** Acting on it caused multiple conflicting instructions to the writer and Production."*
> — `…/memory/feedback_editorial_authority.md`

Compounding it: the Magazine Director **has no "own address" to fall back to** — its identity *is* the unit (`SPRING_AGENT_ID == SPRING_UNIT_ID`), so the suggested fix ("send from your own address") isn't actually possible today. Members coped by hand-signing every message ("— Copy Editor", "— Magazine Director"), and the system prompt even warns them not to trust the envelope `from` — both are signs the platform isn't carrying sender identity for unit-originated messages.

## Issue 4 — Cascading coordination breakdown  ⟶ consequence of 1–3

The agents' own notes read as a catalogue of the fallout, all traceable to the three causes above:

- **Duplicate / divergent briefs.** "Parallel routing by Managing Editor and Magazine Director caused a duplicate brief to reach the Audience Editor on Piece 3." Piece 2 generated two conflicting headline briefs the Audience Editor had to reconcile. (`feedback_routing_protocol.md`, `feedback_audience_editor_briefing.md`)
- **Acting on projected state.** A "running budget" file was being updated out-of-band with optimistic state; the Managing Editor acted on it → "bad readout to the Director, a premature assembly signal to Production, multiple rounds of correction." (`feedback_budget_discipline.md`)
- **Process violation.** Production sent the assembled edition to the publisher without Director review first. (`feedback_production_routing.md`)

The team eventually shipped, but only after burning tens-to-hundreds of corrective messages working around the platform — exactly the confusion you observed.

## Secondary findings

- **60-minute callback JWT.** `SPRING_CALLBACK_TOKEN` (minted by the dispatcher, `CallbackTokenIssuer` / `CallbackTokenOptions.Lifetime`) is `iat + 60min`, injected once at container start and not refreshed. It forces a full container relaunch ~hourly — which, with Issue 2, wipes the workspace each time. Distinct from the per-turn MCP token in Issue 1, but the same "short-lived token vs long-lived persistent agent" mismatch.
- **Inconsistent memory tooling.** Only the Managing Editor wrote `.claude/memory` notes; the others wrote none — and even those notes don't survive teardown. `sv.memory.history_with` is message-history recall, not a durable shared knowledge store.

## Recommended fixes (for triage)

1. **Decouple the MCP session token from the per-turn lifecycle for persistent agents** — issue an agent/session-scoped token (or per-concurrent-turn tokens with per-turn files), and stop revoking a token that concurrent turns still hold. (Issue 1)
2. **Do not reclaim a persistent agent's workspace volume on turn/idle teardown** — persistent agents must keep `spring-ws-<id>` across cycles; only reclaim on genuine decommission. Fix the 500-on-reclaim leak too. (Issue 2)
3. **Carry real sender identity on unit-originated messages** — distinguish "the unit agent" from "a human acting through the unit," and give the unit agent a routable own-address so a member can authenticate who actually sent a directive. (Issue 3)
4. **Provide durable shared team state** — a unit-level status/memory store agents can read/write, so coordination doesn't depend on each agent's wiped local memory. (Issues 2 + 4)
